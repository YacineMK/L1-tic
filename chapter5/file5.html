ACTIVITY TIC

1/système de gestion de version

-logiciel de gestion de version est un logiciel qui permet de conserver un historique des modifications effectuées sur un projet et de revenir a une ancienne version en cas de problème.
-Un système de gestion de version permet de synchroniser le travail de plusieurs développeurs, la connexion aux autres projets et la mise à jour automatique du travail dans le dossier local avec le serveur
-Il existe deux model de système de gestion : model centralisé et un model distribué a:

  • Model centralisé : le code source liée au projet est dans un serveur centrale et les développeurs doivent se connecter pour travailler sur le projet.
  • Model distribué : est l’opposé d’un model centralisé. Le code source est dans un serveur centrale mais tous les développeurs peut télécharger et créer un répertoire locale.

2/Git et GitHub

-Git : est un système de gestion de version décentralisé (model distribué) qui gère les fichiers d’un projet et son historique, ce dernier est appelé repository
-GitHub : est un service en ligne qui joue le rôle de serveur centrale qui héberge le code source et permet a n’importe qui de contribuer à un projet (si le dépôt hébergé est public).

3/la gestion de information selon Git

Git prend des « snapshots » à chaque fois on veut enregistrer ou valider l’état de projet, puis il garde ces snapshots dans une base de donne locale ce qui crée un historique de tous les diffèrent états de projet.
Pour suivre les modifications d’un projet , Un fichier a seaux état : Suivi et non Suivi , Les fichiers suivis sont enregistrés dans la base des données et les fichiers non suivis ne sont pas.
-un fichier suivi lui-même peut avoir trois état :

  • Staged : cette état indique au Git que ce fichier va être dans le nouveau snapshot.
  • Modified : cette état indique au Git que ce fichier a été modifie depuis le dernier snapshot.
  • Commited : tous les fichiers enregistré dans les base de donée ont l’état Commited

4/les zones de travail

Les états de fichiers sont liés à des zones de travail dans Git. Tout projet Git est composé de trois sections : le répertoire de travail (working tree), la zone d’index (staging area) et le répertoire Git (repository).

  • Le répertoire de travail ou “working tree”: une extraction unique (“checkout”) d’une version du projet. Les fichiers sont extraits de la base de données compressée située dans le répertoire Git et sont placés sur le disque afin qu’on puisse les utiliser ou les modifier.
  • La zone d’index ou “staging area”: un simple fichier qui stocke les informations concernant ce qui fera partie du prochain instantané ou du prochain “commit”.
  • le répertoire Git (repository): stocke les méta-données et la base de données du projet.

5/utilisation de Git

Pour utiliser le Git on utilise invite de commande (CMD.)
Apres l’installation de Git . il faut renseigner un nom d’utilisateur et une adresse mail que Git devra utiliser ensuite avec les commandes

  • git config –global user.name
  • git config –global user.mail
Démarrage de depot:
Le dépôt est la copie et l’importation d’un fichier dans le Git . il y a deux façon de crée un dépôt :
- importer un répertoire déjà existant dans Git.
- cloner un dépôt Git déjà existant.
Importation d’un dépôt existant:
Pour importer un répertoire existant on utilise les commandes suivantes :
  • git init (pour initialiser le repertoire)
  • git add (pour indexé des nouveaux fichier)
  • git commit(pour prendre un snapshot des tous les fichier indexés )
Clonage d’un dépôt existant:
après la création d’un dépôt dans Github on utilise l’URL de ce répertoire et les commande suivante pour liée le répertoire avec le serveur
  • git branch -M main
  • git remote add origin https://github.com/Lazy-Skelly/TIC-project.git
  • git push -u origin main
Consultation et restauration de l’historique:
Pour consulter l’historique des versions, l’auteur et date de modification on utilise la commande:
  • git log
Pour restaurer une version ou annuler une modification
  • git restore

6/les branches

Les branches sont des copies de projet due l’utilisateur peut avoir pour travailler sur eux et les modifier sans affecter la copie principale. Cette dernière est appelée branche Master.
Le system de branche utilisé par Git n’utilise pas des copies physiques de tous le projet en revanche la branche est just un pointeur vers un commited.
Note: la branche master n’est pas spécial dans le systéme Git ;elle est traitée de la meme facon que les autres branches
